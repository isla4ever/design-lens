# AstroWind 自动重建实战

本报告验证 Design Lens 是否能在不读取目标项目实现源码的前提下，完成真实长页面的自动捕获、Rebuild Pack 导出、候选实现和量化验收。实验日期为 2026-07-17。

## 实验对象与约束

- 目标：[AstroWind](https://github.com/arthelokyo/astrowind)，MIT License，实验时约 5.8k Stars。
- 页面：AstroWind 首页，桌面文档高度约 8,960px，766 个 DOM 节点。
- 捕获：Design Lens Collector，桌面 `1440x900`、移动端 `390x844`。
- 重建：只使用导出包中的截图、文案、Token、DOMSnapshot、公开资产 URL 和样式证据；不读取 AstroWind 组件或样式源码。
- 验收：同一 Rebuild Pack 和 `npm run verify:rebuild`，阈值为 3% 像素差与 4px 可见节点几何差。

## 捕获结果

| 指标 | 初始实现 | 本轮最终实现 |
| --- | ---: | ---: |
| 长页截图 | 5 段抽样，存在大面积缺口 | 12/12 连续分段成功 |
| 桌面/移动初始与滚动 | 0/4，CDP 动画域错误 | 4/4 成功 |
| 桌面/移动 hover 与 focus | 0/4 | 4/4 成功 |
| 深度样式 | 0 | 104 组 matched/computed styles |
| 响应式 DOMSnapshot | 0 | 桌面、移动各 1 份 |
| 截图限流失败 | 6/12 | 0/12 |
| 捕获后页面滚动 | 未可靠验证 | 可恢复并继续滚动 |
| 捕获耗时 | 7.2s，但证据大量失败 | 15.2s，证据完整度显著提升 |
| 最长任务 | 51ms | 56ms，共 2 次 |

最终包包含 21 个已捕获场景、26 个二进制/JSON 工件、104 组深度样式和 17 个资产线索。请求的 10 个“视口 × 状态”组合中，8 个自动完成；`open` 在桌面和移动端均保持为显式缺口，因为实验没有执行合成点击。

## 本轮发现并修复的问题

1. **真实 MV3 注入失败**：rrweb 依赖产物含 Chrome 拒绝的 Unicode 非字符。构建阶段现在保留运行时语义并转义禁用码点，新增真实扩展注入探针。
2. **Chrome 截图调用配额**：提高长页段数后触发 `MAX_CAPTURE_VISIBLE_TAB_CALLS_PER_SECOND`。后台现在统一串行化截图，并保持 550ms 最小间隔。
3. **失败场景被误报为完整**：普通截图失败没有设置 `document.truncated`。现在任一失败都会标记证据不完整。
4. **Collector 全部深度场景失败**：当前 Chromium 不支持 `Animation.enable`。动画域改为可选，DOM、CSS、截图和响应式证据可独立成功。
5. **伪状态清理使用失效 nodeId**：深度 DOM 读取会刷新前端节点 ID。清理失败时会重新按 selector 定位并清除状态。
6. **移动端初始场景拍在滚动位置**：目标页的 smooth scroll 延迟了恢复。CDP 滚动统一使用 `behavior: "instant"`。
7. **历史导出范围漂移**：打包器曾使用当前全局设置覆盖捕获时的请求。现在场景清单、规格与包内 Brief 优先使用 `rebuildEvidence.request`。
8. **验收覆盖率超过 100%**：重复长页分段被当成独立请求状态。覆盖率现在按请求组合计算，本实验为 8/10，即 80%。
9. **验证器制造动画误差**：未验证动画区域和滚动进入边界缺少有界遮罩。现在只遮罩视口内、面积不超过 50% 且没有 checkpoint 的长动画元素；整屏容器不会被遮罩。
10. **视口外节点误触发几何失败**：几何门禁现在只检查与当前截图相交的节点。

## 验收校准

先使用原站作为候选校准验证器。最终严格策略下：

- 18/21 场景通过；
- 平均像素差 1.33%；
- 可见节点几何失败 0；
- 3 个失败均来自长页中段滚动揭示的非确定过渡帧；
- 覆盖状态仍为 `incomplete`，因为 2 个 open 场景和 motion checkpoint 缺失。

这一步避免把验证器自身的滚动、动画或遮罩错误误判为候选实现问题。

## 证据驱动候选

候选是独立的语义化 HTML/CSS 页面，没有使用截图铺底，也没有读取 AstroWind 源码。第一次严格验收后，只根据报告修正 header 层级、定位上下文、移动端按钮堆叠、首屏高度和总文档高度。

| 指标 | 首轮严格结果 | 一轮结构修正后 |
| --- | ---: | ---: |
| 通过场景 | 6/21 | 9/21 |
| 平均像素差 | 6.62% | 6.25% |
| 可见几何失败 | 27 | 6 |
| 请求状态覆盖 | 80% | 80% |

首屏、部分长页基线和 footer 已接近证据；最大剩余像素热点位于中部 Blueprint、Steps、Blog 模块。移动端初始像素差约 3%，但移动滚动差异约 9.3%。当前证据足以生成有质量的重建草稿，但还不足以支持“完全自动、像素级复刻”的产品承诺。

## 下一阶段优先级

1. **P0：场景级可见节点几何**。每个长页分段采集当前视口的代表节点，而不是只让 CDP 初始场景拥有几何，优先解决中后段模块无法定位的问题。
2. **P0：稳定滚动揭示证据**。为有限动画保存稳定结束态或可寻址 checkpoint；长页分段增加适度重叠，避免进入边界成为证据盲区。
3. **P1：资产策略闭环**。在明确授权下提供受限资产打包、哈希与失败占位策略；`manifest-only` 继续只输出 URL 和用途。
4. **P1：安全 open 状态发现**。只针对明确的 `aria-controls`、`details`、dialog 或 popover 建议自动化计划，默认仍不执行未知业务点击。
5. **P1：网络可复现验收**。记录第三方资源失败并支持本地缓存/替代清单，减少每场景重新加载造成的外部噪声。
6. **P2：按证据健康度约束 Prompt**。缺少几何、资产或 checkpoint 时，Prompt 必须降级为草稿目标，不得声称高保真完成。

## 结论

本轮验证证明“一次操作获得长页、双视口、hover/focus 和深度样式证据”已经可行，而且捕获过程没有产生页面卡死：最长任务为 56ms，捕获结束后页面可继续滚动。真正限制自动复刻精度的已不再是“有没有截图”，而是场景级几何密度、动态状态的确定性和资产可用性。这三项应成为下一阶段主线。
