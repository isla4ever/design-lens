# Bilibili 首页智能捕获与重建实战

本案例验证 Design Lens 在真实动态首页上的两件事：能否在一次智能捕获中得到可复用的结构化证据，以及候选实现能否被同一份证据包量化验收。实验页面为 [哔哩哔哩首页](https://www.bilibili.com/)，实验时间为 2026-07-24。

> [!IMPORTANT]
> 这是经过授权范围约束的工程验证，不是对哔哩哔哩源码、品牌资产或推荐内容的复制。原站截图、封面、Logo、Rebuild Pack 和候选素材只保存在本地 `output/benchmark/bilibili/`，该目录被 `.gitignore` 忽略，不进入开源仓库。

## 验证目标

- 自动取得桌面与移动端的首屏、滚动、悬停和焦点证据。
- 记录 DOMSnapshot、可见几何、匹配样式、交互线索和动效线索，并在页面捕获后继续保持可滚动。
- 使用语义化 HTML/CSS 和 Embla Carousel 8.6.0（MIT）实现独立候选，不把整页截图当作实现。
- 用同一个 Rebuild Pack 重放场景，分别报告像素差、关键节点几何、浏览器错误、状态覆盖和动效关键帧。

## 捕获结果

| 指标 | 结果 | 判定 |
| --- | ---: | --- |
| 智能捕获耗时 | 13.1s | 在 15s 基础预算内完成 |
| 捕获场景 | 13/13 | 桌面/移动初始、滚动、hover、focus 均有截图 |
| 深度样式 | 100 组（68 + 32） | CDP Collector 成功，证据错误 0 |
| 项目节点 | 32 | 关键节点带稳定 `nodeId` 与辅助 selector |
| 页面恢复 | 捕获后可继续滚动 | 无卡死；捕获后滚动到 `y=900` |
| 最长任务 | 130ms | 标记为 `degraded`，没有中断捕获 |
| 动态噪声 | 2,447 次 mutation、4 个长任务 | 只降级证据深度，不阻塞用户页面 |
| 未完成场景 | `open` 桌面/移动 | 没有执行未知业务点击，保持 `planned` |
| 动画基线 | 0 个 seekable checkpoint | 不把静态 CSS/WAAPI 线索冒充为时刻证据 |

捕获状态是 `degraded`，原因是源站持续变化和长任务，而不是采集失败。源站自身产生的第三方请求失败和 console warning 只记录在本地摘要中；候选验收会单独统计候选页面浏览器错误，避免把两个环境混为一谈。

## 候选验收

验收使用固定规则：像素差不超过 3%，关键节点几何差不超过 4 CSS px，状态覆盖必须为 100%，并且候选无浏览器错误。几何契约在四轮迭代中从私有 selector 迁移到 `data-design-lens-node-id`，因此最终报告的几何失败为 0。

| 轮次 | 平均像素差 | 几何失败 | 状态覆盖 | 结论 |
| --- | ---: | ---: | ---: | --- |
| v1 | 35.33% | 15 | 80% | selector 与候选组件命名不一致 |
| v2 | 36.48% | 3 | 80% | 补齐稳定节点定位 |
| v3 | 36.48% | 3 | 80% | 保留差异，未放宽阈值 |
| v4 | 36.53% | 0 | 80% | 几何闭环，像素与场景缺口仍真实可见 |

v4 仍然是 `failed`，这是预期的诚实结果，不是验收器问题。主要差异为：

1. 轮播当前帧、推荐封面和登录提示是随时间/账号状态变化的内容。
2. 顶部导航图标和部分文字是候选的近似语义，不是授权品牌资产。
3. 移动端基线保留了源站的宽布局裁切特征，视觉差异明显，但关键响应式几何仍能命中。
4. `open` 状态和可 seek 的动画 checkpoint 没有证据，不能被 Prompt 或报告描述为已完成。

## 对插件能力的验证

这次实战暴露并验证了几个可作为产品亮点的能力：

- **稳定节点契约**：场景 trigger 同时保存 `nodeId + selector`；候选优先使用 `data-design-lens-node-id`，selector 只做兼容回退。
- **强制 hover 可重放**：验收器设置 `data-design-lens-pseudo="hover"`，不会因为页面浮层挡住真实鼠标而丢失状态。
- **预算与让步**：扫描、样式读取和深度采集按小批次让出浏览器主线程；超预算时降级为部分证据，而不是持续占用页面。
- **动态区域有界遮罩**：只对有明确动效证据、位于当前视口且面积受限的区域遮罩，整屏容器不会被一键忽略。
- **Prompt 与 Skill 同步**：Rebuild 包现在同时输出 `skill.md` 和 `ai-coding-prompt.md`，要求先实现稳定结构，再按验收热点修复。
- **缺口保真**：`planned`、缺失动画时刻和未授权资产继续保留为缺口，避免模型生成“看起来完成”的假复刻。

## 可复现流程

以下命令在项目根目录执行。真实网站捕获需要用户在 Collector 版本中打开目标页面并明确发起智能捕获；候选验收只需要本地候选服务。

```bash
npm ci
npm run build:collector

# 在扩展中完成一次经授权的 Smart Capture，并导出 Rebuild Pack
npm run verify:rebuild -- \
  --pack /path/to/bilibili-rebuild-draft.zip \
  --url http://127.0.0.1:4173
```

本次本地候选页使用 Embla Carousel 8.6.0 的 MIT 发行包，只复用轮播的拖拽、循环、焦点和按钮控制；栅格、导航、搜索热榜、卡片和登录提示均由候选页面自行实现。官方仓库与版本元数据：[Embla Carousel](https://github.com/davidjerleke/embla-carousel)。

## 结论与下一步

Design Lens 已经能稳定捕获真实首页的结构、响应式几何、滚动、hover/focus 和样式证据，并能在高 mutation 页面中结束后恢复用户控制。当前不能声称“自动像素级完美复刻”：真正的剩余瓶颈是动态内容确定性、`open` 状态授权与可寻址动效时刻，而不是缺少截图。

下一阶段应优先补充：

1. 用户明确授权后的 `open` 状态采集，并继续限制在 `aria-controls`、`details`、dialog 或 popover 等可识别目标。
2. CSS/WAAPI 动画的 25%/50%/75% 可 seek checkpoint，且恢复原页面播放状态。
3. 对动态推荐、登录态和第三方资源建立可配置的本地替代/遮罩清单，而不是扩大像素阈值。
4. 将本案例的稳定节点、证据缺口和验收报告继续作为生成 Prompt/Skill 的回归样本。
